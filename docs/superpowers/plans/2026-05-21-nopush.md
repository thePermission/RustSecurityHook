# Per-Project Push Blocker Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `rsh nopush` — an opt-in, per-project flag that blocks `git push`, `gh pr merge`, `glab mr merge`, and `glab mr create` when a `.rsh-nopush` flag file is present in the project root.

**Architecture:** New `src/nopush.rs` module (mirrors `disabled.rs`) owns the flag file and push-command detection. The hook in `main.rs` checks for the flag before dispatching to `run_check`. CLI surface is a new `Nopush` subcommand with an `--off` flag. Self-protection rules in `blacklist.rs` prevent agents from removing the protection.

**Tech Stack:** Rust 2024 edition, `regex` crate (already a dependency), `std::sync::LazyLock`, `tempfile` crate (already in dev-deps).

use regex::Regex;
use std::path::PathBuf;
use std::sync::LazyLock;

static PUSH_RE: LazyLock<Regex> = LazyLock::new(|| {
    Regex::new(
        r"(?:\bgit\b[^|;&\n]*\bpush\b|\bgh\b[^|;&\n]*\bpr\b[^|;&\n]*\bmerge\b|\bglab\b[^|;&\n]*\bmr\b[^|;&\n]*\b(?:merge|create)\b)",
    )
    .unwrap()
});

pub fn flag_path() -> PathBuf {
    PathBuf::from(".rsh-nopush")
}

pub fn is_nopush_active() -> bool {
    flag_path().exists()
}

pub fn is_push_command(cmd: &str) -> bool {
    PUSH_RE.is_match(cmd)
}

        || p.ends_with("/.config/rsh")

fn run_nopush(off: bool) -> ExitCode {
    let flag = nopush::flag_path();
    if off {
        if !flag.exists() {
            eprintln!("rsh: already enabled (push not blocked)");
        } else {
            if let Err(e) = std::fs::remove_file(&flag) {
                eprintln!("rsh: failed to remove flag file: {e:#}");
                return ExitCode::FAILURE;
            }
            eprintln!("rsh: push re-enabled — run 'rsh nopush' to block again");
        }
    } else {
        if flag.exists() {
            eprintln!("rsh: already blocked (push is read-only for this project)");
        } else {
            if let Err(e) = std::fs::write(&flag, "") {
                eprintln!("rsh: failed to create flag file: {e:#}");
                return ExitCode::FAILURE;
            }
            eprintln!("rsh: push blocked for this project — run 'rsh nopush --off' to re-enable");
            if let Err(e) = add_to_gitignore() {
                eprintln!("rsh: warning — could not update .gitignore: {e:#}");
            }
        }
    }
    ExitCode::SUCCESS
}

fn add_to_gitignore() -> anyhow::Result<()> {
    let gitignore = std::path::Path::new(".gitignore");
    let entry = ".rsh-nopush";
    if gitignore.exists() {
        let content = std::fs::read_to_string(gitignore)?;
        if content.lines().any(|l| l.trim() == entry) {
            return Ok(());
        }
        let mut updated = content;
        if !updated.ends_with('\n') {
            updated.push('\n');
        }
        updated.push_str(entry);
        updated.push('\n');
        std::fs::write(gitignore, updated)?;
    } else {
        std::fs::write(gitignore, format!("{entry}\n"))?;
    }
    Ok(())
}
```

```markdown
# ADR 017: Per-Project Push Blocker

**Status:** Accepted  
**Date:** 2026-05-21

## Context

Developers sometimes check out repositories in a read-only intent — even when they hold write access on GitHub/GitLab — to prevent accidental pushes during AI-assisted sessions. No prior `rsh` mechanism covered this: `rsh off` disables all checks, and blacklist rules are global.

## Decision

Add an opt-in, per-project push block via a `.rsh-nopush` flag file. When present, the hook blocks `git push` (all variants), `gh pr merge`, `glab mr merge`, and `glab mr create` with exit code 2. A new `rsh nopush [--off]` subcommand manages the flag and automatically adds `.rsh-nopush` to `.gitignore`.

Self-protection rules (`rsh-nopush-off`, extended `rsh-guard-flag-file`) prevent agents from removing the protection themselves.

## Alternatives Considered

- **Blacklist rules with filesystem check** — would require rules to access filesystem state, breaking the current clean separation between regex matching and environment inspection.
- **`rsh off --push-only`** — semantically confusing (`off` implies everything) and would entangle push-specific logic in the disabled module.
- **Central config by git remote URL** — more powerful but adds complexity and a dependency on `git remote` at hook time; flag file is simpler and consistent with `.rsh-disabled`.

## Consequences

- Flag file must be gitignored (handled automatically).
- No upward directory walk: hook CWD is always the project root when Claude Code invokes it.
- Works for both Claude Code and Codex hooks (both use the same `rsh` hook binary).
```

# Per-Project Push Blocker

`rsh nopush` marks the current project as read-only for push operations. It is opt-in and project-local.

## Activation

```sh
rsh nopush          # block push; creates .rsh-nopush, updates .gitignore
rsh nopush --off    # re-enable push; removes .rsh-nopush
```

## Blocked commands (when .rsh-nopush is present)

| Command | Variants covered |
|---|---|
| `git push` | All flags and arguments |
| `gh pr merge` | All flags |
| `glab mr merge` | All flags |
| `glab mr create` | All flags |

Other git operations (`pull`, `fetch`, `status`, etc.) are not affected.

## Block message

```
rsh blocked push: this project is marked read-only (.rsh-nopush)
hint: run 'rsh nopush --off' to re-enable pushing
```

## Self-protection

Agents cannot run `rsh nopush --off` (rule: `rsh-nopush-off`) or directly delete/rename `.rsh-nopush` (rule: `rsh-guard-flag-file`). Only the developer running `rsh` interactively can re-enable pushing.

## gitignore

`rsh nopush` appends `.rsh-nopush` to `.gitignore` (creates the file if absent). The entry is not removed by `rsh nopush --off` — it is harmless and avoids an extra diff.
```
